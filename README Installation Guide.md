# README Installation Guide

> **Read this first:** the README references two images that are *generated by workflows*, not committed by you. They will 404 until Step 6 — that is expected, not a bug.

## 2. The special repository rule

GitHub renders a README on your profile page **only** from a public repository named exactly the same as your username: **`rahul21896/rahul21896`**.

- This repo **already exists** with an old README, so there is nothing to create.
- Its default branch is **`master`** — do **not** rename it to `main`. The workflows (`snake.yml` push trigger) and the README's raw-image URLs (`.../rahul21896/rahul21896/master/github-metrics.svg`) are wired to `master`.
- The repo must stay **public**; a private profile repo does not render.

## 3. Get the files into the repo

> **Tip:** the repo already exists, so you can do Steps 4 and 5 *before* pushing. If you do, the push-triggered snake workflow succeeds on the first try and you skip one manual re-run.

### Option A — git (recommended)

The local source path contains spaces — keep it quoted exactly as shown.

```bash
# 1. Clone the existing profile repo (default branch: master)
cd "/Users/rahuldhamecha/Projects/Rahul Dhamecha"
git clone git@github.com:rahul21896/rahul21896.git
cd rahul21896

# 2. Copy the new profile files in (README.md replaces the old one)
SRC="/Users/rahuldhamecha/Projects/Rahul Dhamecha/github_profile"
cp "$SRC/README.md" .
cp -R "$SRC/.github" .
cp -R "$SRC/assets" .

# 3. Commit and push to master
git add README.md .github assets
git commit -m "Rebuild profile README with automated stats and workflows"
git push origin master
```

If you use HTTPS instead of SSH, clone with:

```bash
git clone https://github.com/rahul21896/rahul21896.git
```

**After pushing:** the workflows `metrics.yml` and `activity.yml` commit back to `master`. Always `git pull` in this clone before making future edits, or your next push will be rejected as non-fast-forward.

### Option B — GitHub web UI

1. Open <https://github.com/rahul21896/rahul21896> → **Add file → Upload files**. Drag in `README.md` and the `assets` folder, commit directly to `master`. (This replaces the old README.)
2. The `.github` folder is hidden in Finder — either press `Cmd+Shift+.` in Finder to reveal it and drag the whole folder in, or create each workflow by hand: **Add file → Create new file**, type the full path as the filename (e.g. `.github/workflows/snake.yml` — typing `/` creates folders), paste the file contents, commit to `master`. Repeat for all four workflow files.

## 4. Enable Actions write permissions

Three of the four workflows commit back to the repo (snake → `output` branch, metrics → `github-metrics.svg`, activity → README edits). Grant them write access:

1. Go to **github.com/rahul21896/rahul21896 → Settings → Actions → General**.
2. Under **Actions permissions**, select **"Allow all actions and reusable workflows"** — the workflows use third-party actions (`Platane/snk`, `crazy-max/ghaction-github-pages`, `lowlighter/metrics`, `jamesgeorge007/github-activity-readme`, `gautamkrishnar/blog-post-workflow`), so a stricter policy will block them.
3. Under **Workflow permissions**, select **"Read and write permissions"**.
4. Click **Save** (each section saves separately).

## 5. Create the METRICS_TOKEN secret

`metrics.yml` cannot use the built-in `GITHUB_TOKEN` — it needs a **classic Personal Access Token** with `repo` and `read:user` scopes to read account-wide stats. Full PAT creation walkthrough: see **`GitHub Actions Setup.md`**.

1. Create the PAT at **github.com → Settings → Developer settings → Personal access tokens → Tokens (classic) → Generate new token (classic)**, with scopes **`repo`** and **`read:user`**.
2. In the repo: **Settings → Secrets and variables → Actions → New repository secret**.
3. Name: `METRICS_TOKEN` (exact, case-sensitive). Value: the PAT. Click **Add secret**.

Without this secret, only the "GitHub Metrics" workflow fails — everything else runs on the built-in token.

## 6. First run — populate the generated images

Immediately after pushing, two images on your profile show as broken/alt-text. **This is expected:**

| Missing image | Generated by | Lives on |
|---|---|---|
| Contribution snake (light + dark) | `Generate Snake Animation` | `output` branch |
| `github-metrics.svg` ("Detailed Metrics") | `GitHub Metrics` | `master` |

The "Recent GitHub Activity" section also shows its placeholder line until its first run.

Trigger everything once, manually:

1. Open the repo's **Actions** tab.
2. Left sidebar → **Generate Snake Animation** → **Run workflow** → branch `master` → **Run workflow**. (Note: your push in Step 3 already triggered this one — if that run is green, skip the re-run.)
3. Left sidebar → **GitHub Metrics** → **Run workflow**. Takes ~2–3 minutes.
4. Left sidebar → **Recent Activity** → **Run workflow**. Takes under a minute.
5. Leave **Latest Blog Posts** alone — it is intentionally manual-only until you set a real RSS feed in `blog-posts.yml` (see `Customization Guide.md`).

When all runs are green, open <https://github.com/rahul21896> and hard-refresh (`Cmd+Shift+R`). GitHub proxies images through its Camo cache, so freshly generated SVGs can take a few extra minutes to appear.

**If a run fails:** open the run log. The two usual causes are Step 4 not saved (permission-denied on push) and a missing/misnamed `METRICS_TOKEN` (metrics only). Fix, then **Re-run all jobs**.

**After the runs:** `git pull` in your local clone — metrics and activity have now committed to `master`.

## 7. Optional — profile-level settings

- **Include private contributions.** Most of your production work is in private client repos; surface it in the contribution graph, streak, and snake: **github.com/settings/profile → Contributions & activity → check "Include private contributions on my profile"** (also reachable from the **Contribution settings** dropdown on your profile's contribution graph). This shows activity *counts* only — never repo names or code.
- **Bio, location, photo.** On the same settings page, set your location (Bengaluru, India), a one-line bio (e.g. "Senior Full Stack Developer — Laravel | AWS | WordPress | AI"), and a professional photo. These render beside the README.
